## Phase 1: Byte-by-Byte Forensic Audit

I audited the uploaded `update_dusky.sh` as the authoritative baseline. The script is syntactically valid Bash and has a solid high-level architecture: bare Git repo, work-tree collision protection, tracked-change backup/restore, sudo/user execution separation, and self-reexec after updater changes. The major risks are not syntax-level; they are transactionality, partial-failure recovery, non-atomic cross-device moves, and a few places where performance is lost to avoidable subshells and process substitutions. 

### Manifest parsing engine

The original manifest parser is functionally decent but not optimal. `split_manifest_fields` calls `trim` through command substitution for every field, so every manifest line can spawn multiple subshells. The field count logic is correct for `U | script.sh`, `S | ignore-fail | script.sh`, and `U | | script.sh`, but it does extra array construction and extra function calls that Bash 5.3 does not need.

The delimiter model is intentionally strict: one or two pipe delimiters only, with quotes and backslashes hard-blocked in the command field. That is good for deterministic whitespace splitting. The legacy `true script.sh` form is supported, but only `true` is recognized in the command field; `ignore script.sh` is not treated as a legacy command-field ignore even though `ignore` is accepted as a formal flag. That is not a bug if the documented legacy format remains `true`, but it is worth preserving explicitly.

The dynamic `MANIFEST_ARGV_N` arrays are acceptable and fast enough for this workload. They are actually preferable to reparsing command strings at execution time. The only issue is that repeated calls to `parse_update_sequence_manifest` can leave old `MANIFEST_ARGV_N` globals behind if a later manifest is shorter. They are not referenced after the manifest arrays are reset, so this is harmless but untidy.

### Git lock detection and auto-resolution

The original lock logic checks common top-level Git lock files and then uses `find` under `refs` to locate `*.lock` files. It then scans `/proc/[0-9]*/fd/*` and runs `readlink -f` per fd to determine whether a live process still has the lock open. This is reliable enough, but expensive in the rare path and unnecessarily fork-heavy.

The logic correctly refuses to delete an open lock and auto-deletes only locks older than 60 seconds in unattended mode. The remaining race is unavoidable with shell-level tools: a different Git process can create/open a lock between “not open” detection and `rm -f`. The original narrows this but does not eliminate it. The rewrite narrows it further by comparing device/inode/mtime before removal and by using Bash file identity checks (`[[ fd -ef lock ]]`) instead of `readlink -f` per descriptor.

The more serious failure mode is not lock cleanup; it is partial clone. The original `clone_with_retry` clones directly into `$HOME/dusky`. If interrupted or killed mid-clone, the next run sees a present but invalid Git directory and stops as “invalid/corrupted,” requiring manual repair. The rewrite clones into a sibling staging directory and installs it with `mv -T` only after clone success, so partial network failure cannot poison the final bare repo path.

### Collision detection and backup movement

`backup_worktree_collisions_for_ref` is one of the best parts of the original design. It correctly detects:

* exact untracked path collisions with paths tracked by the incoming ref;
* ancestor collisions where a file/symlink blocks a tracked path below it;
* directory collisions, while avoiding unnecessary backup of already-tracked descendants when `honor_current_tracked=true`.

The flaw is atomicity. The original backs collisions up with `mv -- "$coll_src" "$coll_dest"`. If the backup directory is on a different filesystem, GNU `mv` falls back to copy-and-delete semantics. That is not atomic. A crash or full disk during that cross-device move can leave a partial destination and an ambiguous source state. The rewrite requires the active backup root to be on the same filesystem as the work tree, so collision movement remains a same-device rename.

### `backup_user_modifications`

The original captures old mode and object id with `git diff-index --raw --no-renames -z HEAD`, which is the right primitive for deciding later whether a user change can be safely restored over the new upstream tree.

The key flaws are:

1. It copies directly into the final backup path with `cp -a --reflink=auto`. If interrupted mid-copy, a partial path can exist in the backup tree. The manifest is written after copy success, which helps, but the filesystem can still contain ambiguous partial payloads.
2. It uses apparent sizes for directory space estimation, which can undercount metadata-heavy directory trees.
3. If a file exists during manifest capture but disappears before backup, the original records `has_copy=0` while retaining the old non-deletion status. Restoration then skips it rather than preserving the user’s deletion intent.

The rewrite stages each copy under the backup parent and installs it with `mv -T` after copy success. It also promotes vanished paths to deletion intent.

### `atomic_restore_path`

This is the highest-risk function in the original script.

The original sequence is:

1. copy backup to temp under the target parent;
2. move existing target to `tmpdir/.old_$base`;
3. move temp copy into the target path;
4. if the final move fails, attempt rollback.

The critical issue: the displaced original target lives inside `tmpdir`, and `tmpdir` is registered in `CREATED_TEMP_DIRS`. If rollback fails or the process exits at the wrong moment, cleanup can delete the displaced original. That creates a data-loss edge case exactly inside the “atomic restore” path.

The rewrite distinguishes file/symlink replacement from directory replacement:

* file/symlink restore uses `mv -fT` from a temp path in the same parent directory, which is an atomic rename on the same filesystem;
* directory/type-changing restore uses a two-phase move because shell/coreutils cannot atomically replace a non-empty directory with another directory. The old directory is preserved outside the cleanup list if rollback fails.

### Storage and disk-space calculations

The original uses `df -BM` and `df -B1`, parsed with `mapfile`, then separately uses `du -sb --apparent-size` or `stat -c %s`. This works, but it is fork-heavy and can undercount real disk pressure for many small files.

The rewrite uses `stat -f -c '%a %S'` for free bytes and `du -sB1` for backup payload estimates. It also enforces that the active backup root is on the same filesystem as the work tree, which is more important than micro-optimizing the arithmetic.

### Partial failure / interrupted disk write / “bricked” states

The original is vulnerable in three main interruption windows:

1. **Clone interruption**: direct clone into `$HOME/dusky` can leave an invalid final repo.
2. **Reset interruption**: after `git reset --hard`, HEAD may be updated while the work tree is only partially rewritten. On the next run, the original can see `HEAD == upstream` and merely warn about missing/type-mismatched tracked files, leaving the tree partially updated.
3. **Restore interruption**: displaced targets inside cleanup-managed tempdirs can be removed by cleanup after failed rollback.

The rewrite addresses these with atomic clone staging, a sync transaction ledger written before destructive reset/checkout, and safer restore tempdir ownership.

## Phase 2: Modernization & Optimization Strategy

The rewrite replaces the following legacy or fork-heavy Bash patterns:

| Original pattern                                | Replacement                                                         | Reason                                                                   |
| ----------------------------------------------- | ------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| `out+=("$(trim "$field")")`                     | `trim_ref` using nameref mutation                                   | Eliminates per-field subshells.                                          |
| `find "$gitdir/refs" -name '*.lock'`            | Bash `globstar` + `nullglob` over `refs/**/*.lock`                  | Avoids a `find` process in lock detection.                               |
| `readlink -f` per `/proc/*/fd/*`                | Bash `[[ fd -ef lock ]]`                                            | Avoids thousands of `readlink` forks during lock-owner scans.            |
| `df -BM` / `df -B1` parsing                     | `stat -f` byte calculation                                          | Faster, no suffix stripping, no table parsing.                           |
| Direct `git clone` into final dir               | clone into sibling staging dir, then `mv -T`                        | Prevents invalid final repo after network interruption.                  |
| `tee` process substitutions for every subscript | direct log redirection                                              | Removes two `tee` processes per script; faster and no sleep-based flush. |
| Auto-retry hard failures three times            | fail-fast by default, interactive retry only                        | Better idempotency; no blind rerun of partially-applied scripts.         |
| cleanup-managed displaced restore target        | old target preserved outside cleanup in two-phase directory restore | Prevents cleanup from deleting recovery material.                        |

I used Bash 5.3-oriented assumptions where they actually help: namerefs, associative arrays, `globstar`, `printf -v` time formatting, and `bash_source_fullpath` when available. Bash 5.3 introduced `bash_source_fullpath`, and its documented `wait -n` behavior is available, but I did not force `wait -n` into the sudo keepalive because the keepalive waits on exactly one sleep child; ordinary `wait "$pid"` is cheaper and clearer there. ([GNU Mailman Lists][1])

Git reconciliation is made stricter and more idempotent by treating reset/checkout as a transaction:

1. detect and move untracked collisions first;
2. capture tracked user deltas with old mode/object ids;
3. stage backups atomically;
4. write a pending sync transaction file;
5. run `reset --hard` or checkout;
6. restore only safe user changes;
7. clear the transaction file only after successful restoration.

On the next run, an interrupted transaction is detected before normal fetch/reconcile. The script backs up the current tracked tree, completes the reset to the recorded target commit, restores the recorded user modifications if a backup exists, then clears the transaction.

## Phase 4: The Second-Pass Self-Audit

I found and corrected several first-draft regressions before issuing the final artifact.

### Regression 1: `backup_user_modifications` returned failure after successful copy

The function ended with a conditional log:

```bash
(( copied_count == 0 )) && log INFO 'Tracked changes were deletion-only; deletion intent is preserved in the manifest.'
```

When `copied_count > 0`, that expression returns `1`, so under `set -e` a successful backup could look like a failure. Corrected block:

```bash
log OK "Backed up ${#CHANGE_PATHS[@]} tracked change(s) to: $backup_dir"
(( copied_count == 0 )) && log INFO 'Tracked changes were deletion-only; deletion intent is preserved in the manifest.'
return 0
```

### Regression 2: manual-merge free-space failure was masked

The first draft had an `&& ... || true` pattern around manual-merge space checks. Corrected block:

```bash
if (( merge_required_bytes > 0 )); then
    ensure_free_space_for_bytes "$ACTIVE_BACKUP_BASE_DIR" "$merge_required_bytes" 'manual-merge copies' || return 1
fi
```

### Regression 3: pending transaction recovery wrongly required a user-mod backup

Initial clone/checkout transactions can legitimately have no `USER_MODS_BACKUP_DIR`. The recovery path now completes the reset even when there are no user modifications to restore:

```bash
if [[ -n "$USER_MODS_BACKUP_DIR" && -d "$USER_MODS_BACKUP_DIR" ]]; then
    restore_user_modifications || return "$SYNC_RC_UNSAFE"
fi
clear_sync_transaction
```

### Regression 4: sudo initialization failure was masked

The first draft had a compatibility-style `&& init_sudo || true`. Corrected block:

```bash
if [[ "$mode" == S && -z "$SUDO_PID" ]]; then
    init_sudo || { HARD_FAILED_SCRIPTS+=("$script (sudo auth)"); return 1; }
fi
```

### Regression 5: `git diff-index` unexpected errors were treated as clean state

Corrected block:

```bash
if "${GIT_CMD[@]}" diff-index --quiet --ignore-submodules HEAD -- >/dev/null 2>&1; then
    return 0
else
    quiet_rc=$?
    if (( quiet_rc != 1 )); then
        log ERROR "git diff-index failed while capturing tracked changes (rc=${quiet_rc})."
        return 1
    fi
fi
```

### Atomic move review

File and symlink restores are truly atomic at the final path because the staged copy is created under the same parent directory and installed with `mv -fT`. Directory replacement is not truly atomic with portable Bash/coreutils when a non-empty directory is involved. The rewrite treats directory restore as a recoverable two-phase operation and preserves the displaced old directory outside the cleanup list if rollback fails.

### Sudo keepalive FD review

The keepalive subshell closes `LOCK_FD` before entering the sleep/sudo loop:

```bash
[[ -n "${LOCK_FD:-}" ]] && exec {LOCK_FD}>&- 2>/dev/null || true
```

The script does not hold a persistent log fd; it appends to the log file per write. The sudo keepalive does not leak the updater lock into `sleep` or `sudo -n -v`.
